# check-skill

`check-docs` 是一个面向 Markdown 技术资料的交互式质量检查 Skill，适配 Codex CLI、Claude Code CLI 和 OpenCode。

## 两轮工作流

```text
一次性完成全量只读检查并生成稳定的问题编号
  ↓
第一轮：P1 原生交互 → P2 原生交互 → … → 每个问题分别决策
  ↓
第二轮：最终修改计划（含路径和行号）→ 用户确认或调整
  ↓
确认后修改文件并验证 diff
```

为减少等待，Skill 会先完成一次全量只读检查，确定问题总数、稳定编号和当前行号，然后再进入逐问题交互。P1 回答后直接展示已准备好的 P2，不会在每次回答后重新扫描文档；只有检测到文件内容已经变化时才会停止旧流程并重新检查。

第一轮每次只展示一个问题，并在问题正文中显示文件路径和行号。每个问题都可以：

- 采用当前建议。
- 保留当前原文。
- 通过自由输入提供完整的自定义改法。

当前问题必须收到真实回答后才能展示下一个问题。不得把多个问题合并到一次提问中，也不得用“采用全部”跳过逐问题交互。若宿主没有原生提问工具，则每次以一个终端问题结束当前回合，收到输入后再继续。

第二轮不再重复问题原因，只显示最终动作和将要写入的文本。用户确认后才修改文件。

## 行号要求

- 单行问题：`docs/example.md:18`
- 跨行问题：`docs/example.md:18-20`
- 第一轮每个问题和第二轮每个计划项都必须显示路径与行号。
- 第二轮执行前会验证原文仍位于相同位置；文件已变化时不会执行旧计划。

## CLI 交互映射

| CLI | 原生工具 | 使用方式 |
| --- | --- | --- |
| Codex CLI | `request_user_input` | 使用交互式 Plan mode，不使用 `codex exec` |
| Claude Code CLI | `AskUserQuestion` | 使用普通交互会话，不使用 print mode |
| OpenCode | `question` | 使用普通交互会话，不使用无交互运行模式 |

若当前 CLI 未提供原生提问工具，Skill 会退回终端文本输入，但第一轮仍逐问题等待，第二轮仍统一确认。

## 检查范围

- 中文错别字、英文拼写、重复字词、多余空格和明显标点错误。
- 冗余、口语化、语句不通顺和模糊表达。
- 术语、缩略语、大小写、句式和同一概念的一致性。
- OpenHarmony 常用词和术语规范。

代码块、行内代码、URL、邮箱、路径、命令、参数、日志和错误信息默认受保护。

## 测试

使用 `tests/fixtures/review-interaction.md` 测试以下行为：

1. 第一轮依次进行 6 次原生交互，每次只处理一个问题并显示对应行号。
2. 6 个问题都回答后，第二轮显示一张最终计划表和全部行号，不重复完整原因。
3. 第二轮确认前文件哈希不变。
4. 确认后只修改计划中的获批项，代码、URL、路径、命令和日志保持不变。

先运行只读协议校验，不会修改测试文档：

```powershell
python tests/validate_protocol.py
```

然后在 Codex CLI、Claude Code CLI 或 OpenCode 的交互会话中输入：

```text
使用 $check-docs 检查 tests/fixtures/review-interaction.md。第一轮对每个问题分别使用一次原生交互并显示文件路径和行号；第二轮汇总计划供我确认，确认前不要修改文件。
```

第一轮可依次测试不同决定：

```text
P1：采用建议
P2：保留原文
P3：自由输入“该功能支持按配置步骤使用。”
P4：采用建议
P5：保留原文
P6：采用建议
```

第二轮先检查计划表中的位置是否仍为第 5、7、9、11、13、15 行，再选择“确认并执行”或“取消修改”。重复测试前，请用 Git 或备份恢复测试文档，并重新运行只读协议校验确认哈希。
