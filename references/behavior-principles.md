# 总体行为准则

执行任何资料检查前，必须遵守以下原则。

## 只读检查原则

1. 首次检查只读取资料并在对话中报告问题，不修改、格式化或重写任何原文件。
2. 所有问题都必须经过用户决策，包括修改结果唯一的错别字、拼写、重复字词、空格和标点问题。
3. 用户对检查任务的宽泛授权不等于对具体修改的授权。
4. 没有明确决定时，默认保持原文。

## 高置信度原则

1. 仅当能够确认存在问题时才报告。
2. 存在歧义、上下文不足或无法形成安全建议时，不猜测；必要时说明需要补充的信息。
3. 不要为了增加问题数量而扩大检查范围。
4. 不要把规则文件或测试说明中的示例当作用户资料的问题。

## 最小建议原则

1. 建议只覆盖命中规则的最小范围。
2. 不跨句合并或拆分句子，不随意调整信息顺序。
3. 不新增事实，不改变原意，不补充未经证实的技术信息。
4. 同一句存在多个问题时可以给出一个综合建议，但必须说明包含的问题类型。

## 严格不处理

- 代码块和行内代码。
- URL、邮箱、图片链接和 URL 路径。
- 命令、文件路径、参数、占位符和代码标识符。
- 日志、报错信息、自动生成内容和外部引用内容。
- 外部库名、包名、品牌名、产品名和项目固定缩写。
- 中文与英文单词、Markdown 链接之间是否留空格的风格问题。

只有 Skill 内置术语规范适用于当前资料，或用户提供了明确规范时，才能报告相关术语问题。术语问题同样必须由用户决策。

## Markdown 格式保护

1. 保留标题、列表、引用、链接、图片、表格、强调等结构。
2. 可以检查标题和列表项中的正文，但不得更改 Markdown 结构符号。
3. 不把表格分隔符、对齐空格或强制换行空格识别为语言问题。
4. 不对短标题、标签或不完整短语强行添加句号。

## 获批后的修改纪律

1. 修改前再次确认问题编号、用户选择和目标文件当前内容能够对应。
2. 只修改用户选择“修改”或提供了明确自定义文本的项目。
3. 修改后立即检查差异；差异包含未授权内容时，停止并恢复未授权部分。
4. 结合上下文复查修改行，确认 Markdown 结构和技术含义未变化。
5. 复查发现的新问题只报告并重新询问，不自动修改。
