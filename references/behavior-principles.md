# 总体行为准则

执行任何资料检查前，必须遵守以下原则。

# 高置信度原则

1. 仅当能够确认存在问题时才进行修改或报告。
2. 当存在歧义、上下文不足或无法确定修改结果时，保持原样。
3. 不要为了提升表达效果而扩大检查范围。
4. 不要把规则文件中的示例内容当作用户资料的实际问题。

# 最小修改原则

1. 只修改命中当前规则的最小范围。
2. 不跨句合并或拆分句子，不随意调整信息顺序。
3. 不新增事实，不改变原意，不补充未经证实的技术信息。
4. 同一句中存在多个同类低级错误时，可以一次性修复，但不得顺带修改其他类别的问题。

# 严格不处理以下内容

- 代码块和行内代码。
- URL、邮箱、图片链接和 URL 路径。
- 命令、文件路径、参数、占位符和代码标识符。
- 日志、报错信息、自动生成内容和外部引用内容。
- 外部库名、包名、品牌名、产品名和项目固定缩写。
- 中文与英文单词、Markdown 链接之间是否存在空格的风格问题。

只有在 Skill 已配置的术语规范适用于当前资料，或用户提供了明确、已授权的术语规范时，才能依据该规范检查产品名、缩略语和专有词。术语问题只作为待用户确认项，不直接修改。

# Markdown 格式保护

1. 保留标题、列表、引用、链接、图片、表格、强调等 Markdown 格式。
2. 可以检查标题和列表项中的正文，但不得修改 `#`、`-`、`1.` 等结构符号。
3. 不要把表格分隔符、对齐空格或 Markdown 换行空格识别为语言问题。
4. 不要对短标题、标签或不完整短语强行添加句号。

# 修改纪律

1. 修改前检查工作区是否存在用户已有变更，不要覆盖无关内容。
2. 直接修改低级错误后，立即检查差异。
3. 如果修改后可能改变语义或需要额外事实支撑，应恢复原文并转为待用户确认问题。
4. 完成修改后，结合上下文重新阅读修改行，确认 Markdown 结构和技术含义未发生变化。
