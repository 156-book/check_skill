# 检查规则

【规则】用词一致：术语、缩略语、描述同一对象的词汇要全文一致，符合用户的心理预期，禁止使用不同的词汇（包括中英文混用）描述同一事物或操作。

【规则】句式一致：使用一致的句式，使技术文档对外表现出统一风格。例如，统一使用祈使句描述操作。

【规则】概念一致：检测同一概念、术语和专有名词是否始终使用同一表述。例如，检查“用户”和“客户”、“AI模型”和“大模型”是否在表示同一概念时发生混用。

【规则】近义词一致：识别因近义词或同义词交替使用导致的表意模糊。例如，检查“删除”和“移除”、“功能”和“特性”是否在表示同一操作或概念时混用。

【规则】前后一致：发现前后矛盾或形式不一致的描述。例如，前文使用“步骤三”，后文使用“第三步”。

【规则】参考上文：检查当前内容时，可以引用上文作为一致性依据，但不要修改作为参考的上文内容。

【规则】大小写一致：相同英文术语必须与上文采用相同的大小写形式。

【规则】术语表一致：术语和缩略语名称要与 OpenHarmony 术语表保持一致，且全文统一。

【规则】行业术语：OpenHarmony 术语表未涵盖的行业通用术语和缩略语（如 IP、MAC），应与国际、国家或行业标准中的名称保持一致。

【规则】禁止自创缩略语：禁止随意缩写英文单词。除行业通用术语和缩略语外，正文中使用的术语或缩略语应在首次出现或术语表中给出解释。

【规则】缩略语全称：中文文档中，缩略语全称中与缩略语对应的字母应大写。

只有在上下文能够证明两个名称表示同一概念时，才报告一致性问题；无法确定时不要报告。

# OpenHarmony 术语表

## A

### abc文件

方舟字节码（ArkCompiler Bytecode）文件，是 ArkCompiler 的编译工具链以源代码作为输入编译生成的产物，其文件后缀名为 `.abc`。在发布态，abc 文件会被打包到 HAP 中。

### ArkCompiler

方舟编译器，是 OpenHarmony 内置的组件化、可配置的多语言编译和运行平台，包含编译器、工具链、运行时等关键部件，支持高级语言在多种芯片平台的编译与运行，并支撑 OpenHarmony 标准操作系统及其应用和服务运行在手机、个人电脑、平板、电视、汽车和智能穿戴等多种设备上。

### ArkTS

OpenHarmony 生态的应用开发语言。它在 TypeScript（简称 TS）的基础上，扩展了声明式 UI、状态管理等能力，让开发者可以以更简洁、更自然的方式开发应用。

### ArkUI

OpenHarmony 原生 UI 框架。一套极简、高性能、跨设备应用设计研发的 UI 开发框架，支撑开发者高效地构建跨设备应用 UI 界面。

## B

### BMS

Bundle Manager Service，包管理服务。

## C

### CES

Common Event Service，OpenHarmony 中负责处理公共事件的订阅、发布和退订的系统服务。

## D

### DMS

Distributed Management Service，分布式管理服务。

## E

### ExtensionAbility

Stage 模型中的组件类型名，即 ExtensionAbility 组件，提供特定场景（如卡片、输入法）的扩展能力，满足更多使用场景。

## F

### FA

Feature Ability，在 FA 模型中代表有界面的 Ability，用于与用户进行交互。

### FA模型

API version 8 及更早版本支持的应用模型，已经不再主推。建议使用新的 Stage 模型进行开发。

## H

### HAP

Harmony Ability Package。一个 HAP 文件包含应用的所有内容，由代码、资源、三方库及应用配置文件组成，其文件后缀名为 `.hap`。

### HCS

HDF Configuration Source，是 HDF 驱动框架的配置描述语言。该语言用于实现配置代码与驱动代码解耦，以及便于配置管理，是一种以 Key-Value 为主体的文本格式。

### HC-GEN

HDF Configuration Generator，是 HCS 配置转换工具，可以将 HDF 配置文件转换为软件可读取的文件格式。

### HDF

Hardware Driver Foundation，硬件驱动框架，用于提供统一外设访问能力和驱动开发、管理框架。

### Hypium

Hyper Automation 与 ium 的组合词，是 OpenHarmony 自动化测试框架名称，以超自动化测试为理想目标，ium 意指稳定、可靠的测试框架能力底座。

## I

### IDN

Intelligent Distributed Networking，是 OpenHarmony 特有的分布式组网能力单元。开发者可以通过 IDN 获取分布式网络内的设备列表和设备状态信息，以及注册分布式网络内设备的在网状态变化信息。

## P

### PA

Particle Ability，在 FA 模型中代表无界面的 Ability，主要为 Feature Ability 提供支持，例如作为后台服务提供计算能力，或作为数据仓库提供数据访问能力。

## S

### Service widget，服务卡片

将用户应用程序的重要信息以服务卡片的形式展示在桌面，用户可通过快捷手势使用卡片，以达到服务直达、减少层级跳转的目的。

### Stage模型

API version 9 开始新增的应用模型，提供 UIAbility、ExtensionAbility 两大类应用组件。由于该模型还提供 AbilityStage、WindowStage 等类作为应用组件和 Window 窗口的“舞台”，因此称为 Stage 模型。

### Super virtual device，超级虚拟终端

亦称超级终端，通过分布式技术将多个终端的能力进行整合，存放在一个虚拟硬件资源池中，根据业务需要统一管理和调度终端能力，对外提供服务。

### SysCap

全称 System Capability，即系统能力。不同值用于指代 OpenHarmony 中各个相对独立的特性或系统能力，如蓝牙、Wi-Fi、NFC 等。每个特性或系统能力对应多个 API，每个 API 定义上包含相应的 SysCap 标签。

### System Type，系统类型

- **Mini System，轻量系统**：面向 MCU（Microcontroller Unit，微控制单元）类处理器，例如 ARM Cortex-M、RISC-V 32 位设备，资源极其有限，参考内存 ≥128 KiB，提供丰富的近距连接能力和外设总线访问能力。典型产品有智能家居领域的联接类模组、传感器设备等。
- **Small System，小型系统**：面向应用处理器，例如 Arm Cortex-A 设备，参考内存 ≥1 MiB，提供更高的安全能力、标准图形框架和视频编解码等多媒体能力。典型产品有智能家居领域的 IPCamera、电子猫眼、路由器以及智慧出行领域的行车记录仪等。
- **Standard System，标准系统**：面向应用处理器，例如 Arm Cortex-A 设备，参考内存 ≥128 MiB，提供增强的交互能力、3D GPU、硬件合成能力、更多控件和更丰富的图形动效，并提供完整的应用框架。典型产品有高端冰箱显示屏等。

## U

### UIAbility

Stage 模型中的组件类型名，即 UIAbility 组件，包含 UI，提供展示 UI 的能力，主要用于和用户交互。

# 输出要求

1. 术语和表达一致性问题只作为待用户确认项，不直接修改。
2. 必须注明文件路径、当前行号、存在问题的原文和作为依据的上文或术语规范。
3. 原因中应具体说明混用了哪些术语、缩略语、大小写或句式。
4. 给出最小修改建议，保证同一概念的写法一致。
5. 多条问题按文件和原文出现顺序排列。
6. 保留 Markdown 格式，不处理代码块、URL、命令、路径和代码标识符。
7. 不输出 JSON，使用以下自然语言格式：

```text
文件路径:行号
原文：存在不一致问题的句子
参考：作为依据的上文句子或 OpenHarmony 术语规范
原因：说明具体的不一致问题
建议：保证用词、术语或句式一致的最小修改文本
```
