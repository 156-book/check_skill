# 两轮交互流程

本流程把“轮”定义为工作阶段：第一轮可包含与问题数量相同的多次交互，第二轮只汇总和确认一次。写文件前始终保持只读。

## 通用约束

1. 使用稳定编号 `P1`、`P2`……，两轮中编号含义不得改变。
2. 单行位置写为 `path/to/file.md:12`；跨行写为 `path/to/file.md:12-14`。
3. 行号以第一轮读取的当前文件为准。第二轮确认前验证原文仍在相同位置；不一致时停止并重新检查。
4. 第一轮一个交互只处理一个 ID；第二轮一张计划表统一确认。
5. 第一轮和第二轮确认前不得调用编辑工具、格式化器或提交命令。
6. 问题按文件分组，并在每个文件内按原文出现顺序排列；同一句多个问题可合并为一个 ID，但必须列全类型和原因。

## CLI 原生工具映射

先检查当前工具列表，只调用真实存在且当前主线程可调用的工具。

- **Codex CLI**：在交互式 Plan mode 中使用 `request_user_input`，每次只传入一个问题对象。
- **Claude Code CLI**：使用 `AskUserQuestion`，每次只询问一个问题。
- **OpenCode**：使用 `question`，每次只询问一个问题。

不要猜测工具参数，遵守宿主实际暴露的 schema。不要在 `codex exec`、Claude Code print mode 或其他非交互执行模式中尝试本流程。

原生工具不可用时，退回普通终端输入：第一轮每次仅输出一个问题并结束当前回合，收到回答后再展示下一个；第二轮仅输出一次计划确认。不要打印伪造的工具 JSON，也不要在同一次输出中假定用户已经回答。

## 第一轮：逐问题交互

### 展示

按文件和行号排序。每次只展示一个问题，包含：

| 字段 | 要求 |
| --- | --- |
| 进度 | 例如 `P2（2/6）` |
| 位置 | 文件路径和当前行号或行号范围 |
| 类型 | 一个或多个明确问题类型；合并时逐项列全 |
| 原文 | 能定位问题的最小原文范围 |
| 原因 | 具体命中的规则 |
| 建议 | 可直接写回的完整文本；无法安全建议时说明缺少的信息 |

### 交互

展示后立即为当前 ID 调用一次原生提问工具。问题正文再次包含 ID 和位置，例如：

```text
P2 · docs/a.md:20：如何处理这个问题？
```

有安全建议时提供两个显式选项，并保留宿主的自由输入入口：

1. `采用建议 (Recommended)`：记录建议文本，暂不写文件。
2. `保留原文`：记录不修改。
3. `自由输入/其他`：用户输入完整替换文本，记录为用户自定义。

没有安全建议时，将 `保留原文 (Recommended)` 作为首选；用户可在自由输入中补充信息或输入完整替换文本。不要提供不可执行的“采用建议”。

必须等待当前工具返回后才能展示下一个问题。不要在一个工具调用中放入多个 ID，不要提供“采用全部”或“全部保留”替代逐问题决策。用户中断时保持文件不变。

## 第二轮：计划确认

### 展示

全部问题都有决定后输出一张最终计划表：

| 字段 | 要求 |
| --- | --- |
| ID | 沿用第一轮编号 |
| 位置 | 再次显示文件路径和行号或行号范围 |
| 动作 | 采用建议、用户自定义、保留或待补充 |
| 最终文本 | 将实际写入的完整文本；不修改时写“不修改” |

不要重复完整原因。计划过长时可按文件分组，但仍只调用一次确认工具。

### 交互

调用一次原生提问工具，问题为“是否按上表执行修改计划？”：

1. `确认并执行 (Recommended)`：允许写入计划中的获批项。
2. `取消修改`：不写文件并结束。
3. 使用自由输入入口按编号调整，例如 `P1=保留；P3=改为：……`。

收到调整时，只更新受影响的计划项并在第二轮内再次确认；不返回第一轮。没有收到“确认并执行”时，写入权限始终关闭。

## 确认后的执行

1. 再次验证路径、原行号和原文。
2. 只写入已确认的采用或自定义项。
3. 不执行计划外的语法优化、句式调整、术语替换或表达润色，保留所有原始 Markdown 格式。
4. 检查 diff，并映射到对应 ID。
5. 按文件和问题类型统计实际修复处数；合并 ID 中的每个实际修复类型分别计数。
6. 输出包含修复统计的简短结果摘要，不开启第三轮。

## 失败关闭

- 工具仅根线程可用时，由主线程执行交互；子代理不得向用户索取决定。
- 文件内容在两轮间发生变化时，不执行旧计划。
- 用户输入含义不清时，只澄清当前 ID，不做猜测性修改。
- 用户中断、未回答或取消时，保持文件不变。
